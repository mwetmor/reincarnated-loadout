§ 1 Three-machine architectural overview

The Reincarnated project has outgrown single-machine development. Three distinct machines, each playing to its hardware/software strengths, connected by a Pi-hosted middleware layer.

CONTENT ENGINE

Mac (8GB RAM)

Python content engine. Agent orchestration via CLI. Design work + canonical doc authorship. JSON output generation from engine pipeline runs. Git commits. Vercel deploys for loadout app.

Strengths consumed: high developer productivity, mature Python toolchain, agent CLI affordances, Git velocity.

Constraints respected: 8GB RAM insufficient for Unreal Engine; no Windows-native UE5 tooling.

MIDDLEWARE — ALWAYS ON

Pi 5 (8GB / 128GB SD)

Samba file share (Phase 1). FastAPI HTTP service (Phase 2). PostgreSQL host (existing per D1 ratification 2026-05-25). Always-on; both Mac and PC interact via the Pi rather than direct Mac↔PC communication.

Strengths consumed: low power consumption, always-on operation, headless deployment, network-resident services.

Constraints respected: 8GB RAM modest for FastAPI + PostgreSQL but adequate for solo-dev workload; SD card I/O sufficient for JSON file serving.

GAME ENGINE

PC (specs TBD — § 9)

Unreal Engine 5 development. Game runtime integration. Character creation system implementation. Consumes engine outputs from Pi shared folder (Phase 1) + via HTTP API (Phase 2).

Strengths required: Windows-native UE5 tooling, RTX-class GPU for Lumen + Nanite + Niagara, ample RAM for engine work, NVMe SSD for Unreal install + content.

Status: PROCUREMENT GATED — see § 9 for detailed spec discussion.

1.1 The augmentation-not-replacement principle

The Pi is additive infrastructure for the Mac↔PC seam, NOT a replacement of the existing Mac↔Vercel flow. Content takes the route that matches its consumption pattern:

Content typePathRationale
Canonical docs, design artifacts, codeMac → Git → distributedVersioned, distributed, multi-agent; existing pattern
Loadout app data (class JSONs, manifests)Mac → Git → VercelVite glob auto-discovery; SPA served to web users
Engine-output artifacts for Unreal consumptionMac → Pi share → PC readsNEW — Pi's primary role for Phase 1
§ 12 hero+gear Meshy artifactsMac → Pi share → PC readsNEW — composes naturally with Phase 2 Unreal integration
PostgreSQL engine-internal dataPi (already)D1 ratified 2026-05-25; continues
Telemetry from simPi PostgreSQLQueryable, structured; better than file-based for analytics
Research artifacts (legolas)Mac → GitMulti-agent consumption via git checkout; no Pi routing needed
Core principle

No central Pi monopoly on the project's data. Pi handles the new Mac↔PC seam; everything else stays on its existing path. This decouples Pi rollout risk from the rest of the project — if Phase 1 hits issues, the loadout app + canonical docs + design workflow are unaffected.

§ 3 Network architecture

The three machines must share a LAN. Network reliability is load-bearing for the iteration workflow.

3.1 Network topology requirement

All three machines (Mac, Pi, PC) connected to the same LAN. Same subnet. No NAT or VPN traversal required. Typical home network setup:

Internet
   │
   └── Modem/Router
         │
         ├── (wired) Pi 5         ← recommended: wired ethernet for reliability
         ├── (wired) PC           ← recommended: wired ethernet for Unreal asset transfer speed
         └── (WiFi)  Mac mini     ← acceptable: WiFi sufficient for JSON file sizes; USB ethernet upgrade if flaky

3.2 Mac connection method — WiFi vs wired ethernet

Mac connection decision

Default: WiFi. Mac mini typically has built-in WiFi but no built-in ethernet (depends on model). For solo dev iteration with JSON file sizes typically < 5MB per artifact, WiFi 802.11ac is sufficient.

Upgrade trigger: if Mac → Pi share access proves flaky (frequent reconnections, slow file write/read, intermittent timeouts), add USB-C to gigabit ethernet adapter ($15-30). Plug into router; configure macOS network preferences to prefer ethernet.

3.3 Pi connection method — wired ethernet recommended

Pi network reliability

The Pi is the always-on middleware. If Pi disconnects from the network, both Mac and PC lose access to shared infrastructure. Wired ethernet recommended over WiFi for reliability. Plug ethernet cable from Pi to router. WiFi acceptable as fallback if cable routing impractical.

3.4 Address resolution — static IP vs .local mDNS

Two options for how Mac and PC find the Pi on the LAN:

OptionProsConsRecommended?
.local mDNS (e.g., raspberrypi.local or custom hostname like reincarnated-pi.local)Zero setup; built-in on Pi + macOS; Bonjour installable on WindowsOccasionally flaky depending on router; some routers block mDNS broadcastsSTART HERE — simplest path; upgrade only if flaky in practice
Static IP via router DHCP reservation (e.g., 192.168.1.50 reserved for Pi MAC address)More reliable; predictable address; survives router DNS quirksRequires router admin access + DHCP reservation config; address changes if router replacedUpgrade option if .local unreliable
Static IP via Pi-side config (set in /etc/dhcpcd.conf)Pi controls own address; router-independentRisk of address conflict with other DHCP devices; harder to debug if Pi misconfiguresNot recommended — DHCP reservation cleaner

3.5 Windows + .local resolution

Windows mDNS gotcha

Windows historically didn't support mDNS / Bonjour out of the box. Modern Windows 10/11 should handle raspberrypi.local via built-in support, but if PC can't resolve, install Apple Bonjour Print Services from Apple (free; included with iTunes for Windows historically; standalone download available). This makes .local resolution work consistently from Windows.

3.6 Recommended Pi hostname

Default Pi OS hostname is raspberrypi. Recommend changing to a project-specific hostname:

# On Pi, run:
sudo hostnamectl set-hostname reincarnated-pi
sudo reboot
# Verify after reboot:
hostname  # should print: reincarnated-pi

Mac and PC will then access via reincarnated-pi.local — clearer than generic raspberrypi.local if multiple Pi devices exist on network.

§ 4 Phase 1 — Samba file sharing (exhaustive)

Goal: Mac engine outputs accessible to PC via Pi network share. ~4-8 hours focused execution session. Phase 1 is the foundation; subsequent phases extend.

Phase 1

Samba file sharing

Pi exposes shared folder via Samba. Mac mounts share via Finder + macOS SMB client. PC mounts share via File Explorer + Windows SMB client. Engine outputs flow Mac → Pi → PC for Unreal consumption.

4.1 Pre-flight checklist

Before starting Phase 1 work

  • Pi powered on, ethernet connected, accessible via SSH (verify: ssh matt@reincarnated-pi.local from Mac)
  • Pi PostgreSQL service running (verify: systemctl status postgresql on Pi)
  • Mac can reach Pi via mDNS (verify: ping reincarnated-pi.local from Mac terminal)
  • SD card has at least 5GB free for new install + working space
  • Mac firewall not blocking SMB ports 139/445 (typically not blocked by default)
  • Plan ~4-8 hour focused window; partial execution can leave system in awkward state
  • Mac local clone of collab repo current; commit any in-progress work before starting

4.2 Step-by-step Pi setup

1
Update Pi OS

Ensure Pi OS package list current + system updated.

sudo apt update
sudo apt upgrade -y
2
Install Samba 
sudo apt install samba samba-common-bin -y

Samba service installs but won't be configured yet.

3
Create the shared folder with explicit path

Per canonical migration-readiness principle, use explicit path NOT default Samba locations:

sudo mkdir -p /home/matt/data/shared
sudo chown matt:matt /home/matt/data/shared
chmod 755 /home/matt/data/shared

Also create subdirs for clarity:

mkdir -p /home/matt/data/shared/engine-output
mkdir -p /home/matt/data/shared/visual-artifacts
mkdir -p /home/matt/data/shared/meshy-handoff
4
Set Samba user password for the matt account

Samba auth is independent of Pi OS auth. Set a Samba-specific password:

sudo smbpasswd -a matt
# Enter password twice; this is the Samba auth for Mac+PC access

The username matt and password set here are what Mac/PC will use to connect — independent of Pi OS user password.

5
Edit smb.conf to expose the share

Open with sudo:

sudo nano /etc/samba/smb.conf

Scroll to bottom; add new share definition:

[reincarnated]
   path = /home/matt/data/shared
   browseable = yes
   writable = yes
   only guest = no
   create mask = 0664
   directory mask = 0775
   valid users = matt
   # Suppress macOS metadata noise:
   veto files = /.DS_Store/.AppleDouble/.AppleDB/.AppleDesktop/Network Trash Folder/Temporary Items/TheVolumeSettingsFolder/
   delete veto files = yes
   # Suppress Windows thumbnail cache noise:
   veto files = /Thumbs.db/
   # SMB3 protocol (modern; macOS deprecates older SMB versions):
   # Note: smb.conf protocol selection lives in [global] not per-share

Also ensure [global] section sets SMB3 minimum:

# In [global] section, add or verify:
   server min protocol = SMB2_10
   server max protocol = SMB3
   # Mitigates legacy SMB1 vulnerabilities; macOS prefers SMB3

Save (Ctrl+O, Enter, Ctrl+X in nano).

6
Test smb.conf syntax 
testparm

Should print parsed config with no errors. Press Enter when prompted to dump service definitions.

7
Restart Samba service 
sudo systemctl restart smbd
sudo systemctl enable smbd  # auto-start on boot

Verify running:

systemctl status smbd
8
Confirm port 445 (SMB) listening 
sudo ss -tlnp | grep 445
# Should show smbd listening on port 445

6.2 Migration steps

1
Connect SSD to Pi USB 3 port

Use blue (USB 3) port, not black (USB 2). Verify recognition:

lsblk
# SSD typically appears as /dev/sda or /dev/sdb
2
Partition + format SSD with ext4 
sudo fdisk /dev/sda

sudo mkfs.ext4 /dev/sda1
sudo e2label /dev/sda1 reincarnated-data
3
Mount at /data 
sudo mkdir -p /data
sudo mount /dev/sda1 /data

4
Add to /etc/fstab for permanent mount

Get UUID:

sudo blkid /dev/sda1
# Note the UUID value

Edit fstab:

sudo nano /etc/fstab

5
Stop services before data move

sudo systemctl stop smbd

§ 7 Backup strategy (exhaustive)

SD card can fail without warning. Backup discipline is non-negotiable from Day 1, even for the "temporary" SD card phase. Per canonical risk register R1.

7.1 Day-1 backup minimum

Non-negotiable

Weekly automated rsync of Pi's /home/matt/data + PostgreSQL pg_dump to Mac local disk. Period.

7.2 Backup script (Pi-side)

Author /home/matt/bin/backup.sh:

#!/bin/bash
# Pi-side backup script — run nightly via cron

BACKUP_DIR=/home/matt/data/backup
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
LOG=$BACKUP_DIR/backup.log

mkdir -p $BACKUP_DIR

echo "[$TIMESTAMP] Backup starting" >> $LOG

# PostgreSQL dump
pg_dump -U matt -h localhost reincarnated > $BACKUP_DIR/postgres_${TIMESTAMP}.sql
echo "[$TIMESTAMP] pg_dump complete: $(ls -la $BACKUP_DIR/postgres_${TIMESTAMP}.sql)" >> $LOG

# Retention: keep last 14 days of pg_dumps
find $BACKUP_DIR -name "postgres_*.sql" -mtime +14 -delete

echo "[$TIMESTAMP] Backup complete" >> $LOG

Make executable + add to cron:

chmod +x /home/matt/bin/backup.sh
crontab -e
# Add line:
0 2 * * * /home/matt/bin/backup.sh
# Runs nightly at 2am

7.3 Mac-side weekly rsync pull

On Mac, author ~/bin/weekly-pi-backup.sh:

#!/bin/bash
# Mac-side weekly rsync from Pi

BACKUP_DEST=~/reincarnated-pi-backups
TIMESTAMP=$(date +%Y%m%d_%H%M%S)

mkdir -p $BACKUP_DEST

rsync -avz --delete matt@reincarnated-pi.local:/home/matt/data/ $BACKUP_DEST/current/

# Snapshot the current backup state weekly
tar -czf $BACKUP_DEST/snapshot_${TIMESTAMP}.tar.gz -C $BACKUP_DEST current/

# Retention: keep last 8 weekly snapshots
ls -t $BACKUP_DEST/snapshot_*.tar.gz | tail -n +9 | xargs rm -f 2>/dev/null

echo "Weekly Pi backup complete: $BACKUP_DEST/snapshot_${TIMESTAMP}.tar.gz"

Schedule via Mac launchd or cron. Sundays at noon recommended (low-activity window).

7.4 Test restore (non-negotiable)

Test the restore at least once before declaring backup discipline "operational"

An untested backup is hope, not infrastructure. Day-1 task: verify backup restores.

Test restore steps:

# Create a test database from a backup:
psql -U matt -h localhost -c 'CREATE DATABASE reincarnated_restore_test;'
psql -U matt -h localhost -d reincarnated_restore_test < /home/matt/data/backup/postgres_<timestamp>.sql

# Query something:
psql -U matt -h localhost -d reincarnated_restore_test -c 'SELECT COUNT(*) FROM <some_table>;'

# Compare counts to production:
psql -U matt -h localhost -d reincarnated -c 'SELECT COUNT(*) FROM <some_table>;'

# If counts match, backup is restorable. Drop test:
psql -U matt -h localhost -c 'DROP DATABASE reincarnated_restore_test;'

LAN-only infrastructure. Solo-dev project. Reasonable security without enterprise overhead.

8.2 Hardening recommendations

Day-1 hardening

  • SSH key auth from Mac to Pi; disable password SSH after key works (PasswordAuthentication no in /etc/ssh/sshd_config)

  • Pi firewall (ufw) configured to allow only LAN subnet for SMB + FastAPI + PostgreSQL ports
  • Verify router does NOT forward any ports to Pi
  • Verify router admin password is not default

9.2 Three immediate hardware actions (per Matt 2026-05-30 Claude PC spec report)

Three actions before sustained Pi infrastructure work

Per Claude's spec report § 5 priority order, three hardware-side actions land before Phase 1 Pi integration cycles for stability + performance + reliability:

1
Enable XMP / EXPO profile in BIOS

DDR5 modules rated 5600 MT/s but currently running ~4000 MT/s — XMP/EXPO disabled. Reboot into BIOS (DEL key during POST on MSI); navigate to OC / Memory section; enable XMP Profile 1 (or EXPO if listed). Reclaim DDR5-5600 speed.

Impact: Free performance for Unreal asset streaming + cook performance + iteration loop responsiveness. Matters more when 64GB RAM upgrade lands.

Time: ~10 min including reboot verification

2
Update motherboard BIOS

Current BIOS dates to Jan 2024; predates Intel's 13th/14th-gen voltage instability microcode fixes (0x129 / 0x12B from mid-to-late 2024). i7-14700F is exactly the chip family affected. Sustained heavy compiles + Unreal cooks + lightmap bakes are the workloads that exposed the defect originally.

Process: Download latest BIOS for MSI PRO B760-VC WIFI 7 from MSI support site; flash via USB drive per MSI's documented procedure; verify post-flash with HWiNFO or similar to confirm microcode revision.

Time: ~30-45 min including download + flash + verification

Risk: BIOS flash carries small bricking risk; backup current settings; ensure stable power; don't interrupt the flash process. Read MSI's specific guidance carefully.

3
Connect wired Ethernet from PC to router

Wi-Fi 7 at 1.2 Gbps is fast but variable in latency. For the Mac↔Pi↔PC pipeline (continuous JSON + 3D model transfers via Pi shared folder), wired ethernet is more deterministic. The PC's onboard NIC is present but disconnected (showing link-local 169.254.x.x address).

Process: Run ethernet cable from PC to router; Windows auto-configures via DHCP; verify with ipconfig that the ethernet interface gets a 192.168.1.x address; optionally disable Wi-Fi to force ethernet preference.

Time: ~5-15 min depending on cable routing distance

11.1 Critical iteration considerations

Atomic file writes

When engine writes JSON to shared folder, write to temporary file first, then rename to final filename. Prevents Unreal from reading partially-written files. Standard pattern:

# Python pattern in engine emit code:
import os
with open(f'{path}.tmp', 'w') as f:
    json.dump(data, f, indent=2)
os.rename(f'{path}.tmp', path)
# Rename is atomic from reader perspective

Schema versioning

Every JSON file includes manifest_version or schema_version field. Unreal reader checks before parse. Prevents Unreal from crashing on unexpected JSON structure.

File naming conventions

Include identifying information in filenames: character ID, iteration counter, timestamp. Tracks which version produced which Unreal behavior.

# Good:
character_form_021_v3_20260530.json

# Bad (ambiguous):
output.json

Logging on both sides

Engine logs what JSON it generated (with version, timestamp, source data). Unreal logs what it imported (with version, timestamp, any parse errors). When iteration produces unexpected results, logs on both sides enable diagnosis.

§ 13 Implementation hurdles checklist

Specific hurdles to anticipate during execution, in rough probability order.

Phase 1 likely hurdles

  • macOS deprecating SMBv1/v2 — modern macOS prefers SMB3; ensure smb.conf sets server min protocol = SMB2_10 minimum
  • Windows mDNS resolution flaky — install Apple Bonjour Print Services as fallback; or use static IP if cleaner
  • Mac WiFi flaky for share access — USB ethernet adapter $15-30 fixes; recommend at first sign of flakiness
  • Pi SD card I/O slow on first Samba use — typical first-mount caching; subsequent ops fast; not a hurdle, just expectation
  • macOS .DS_Store noise — smb.conf veto files directive suppresses; documented in § 4.2 step 5
  • Permission propagation between macOS and Pi — Samba force user = matt directive solves; documented in § 12 F14
  • Forgetting to enable systemd autostart — sudo systemctl enable smbd after install; verify with reboot test
  • NTP not synced between machines — timestamps drift; backup retention buggy; documented in § 12 F13

Phase 2 likely hurdles

  • Python venv path confusion — explicit absolute paths in systemd ExecStart; documented in § 5.2 step 6
  • PostgreSQL connection refused — verify pg_hba.conf allows localhost connections; reload sudo systemctl reload postgresql
  • FastAPI service crashes on first request — check journalctl logs: sudo journalctl -u reincarnated-api -n 100
  • Unreal VaRest plugin not finding HTTP service — verify firewall allows 8000; verify service binding 0.0.0.0 not 127.0.0.1
  • Schema version mismatch between engine + Unreal — document MIGRATION.md per emit; Unreal reader validates before parse

Phase 3 likely hurdles

  • SSD not detected on first connect — try other USB 3 port; lsblk verify; possibly USB cable issue
  • SSD power-budget issues — if Pi USB ports brownout, use powered USB hub or smaller SSD or SSD with external power
  • PostgreSQL data_directory move requires PG restart in maintenance mode — stop service before copy; update conf; restart
  • UUID-based fstab entry stale after SSD replacement — update with new UUID via blkid
  • Symlink confusion if mixing /home/matt/data and /data paths — clear documentation; choose one canonical path

PC-specific likely hurdles

  • Unreal install failures on Windows — Epic Games Launcher install logs at %LOCALAPPDATA%\EpicGamesLauncher\Saved\Logs\
  • Antivirus interfering with UE5 compile — exclude UE5 install dir + project dir from real-time scanning
  • GPU driver outdated for Lumen — NVIDIA Studio driver recommended for UE5; install latest
  • Insufficient RAM for large project loads — 16GB will struggle; 32GB target
  • Windows network discovery slow — Pi share via static IP usually faster than .local on Windows; SMB protocol negotiation occasionally takes a few seconds first time
