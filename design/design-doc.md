# Reincarnated Loadout — Design Document

**Status:** v0 locked decisions. Source of truth for this repository.

---

## Vision

The Reincarnated Loadout App is an **analytical inspection tool** for character builds generated by the Reincarnated engine. In its current form (v0), it lets a player examine a single character class: browse the skill tree, allocate skill points, read stat distributions, and understand what their class is actually good at.

The **North-Star use case** that drives every architectural choice: *"I body-swapped into a hybrid slime-geomancer at L23 — who else is running this build?"* A player lands in an unfamiliar body mid-season. They need to:

1. Identify what archetype/role they're in (Page 1: Loadout)
2. Understand how this class has historically performed and what other players have run (Page 2: Analytics)
3. Eventually share their exact build with a community and find doppelgangers (v1+: Sharing)

This is why the analytics page is described as "super important" — it's the first step toward community context. The sharing stub in v0 preserves design intent for v1.

---

## Locked Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Two pages: Loadout (P1) + Analytics (P2) | P1 = "what am I"; P2 = "how has this class/archetype historically played". Both needed before sharing is meaningful. |
| 2 | Single-scroll desktop, tabs/swipe mobile | Desktop users get full context without context-switching. Mobile users work within viewport constraints. |
| 3 | Skill tree: two-step interaction (tap → details panel → confirm) | Prevents accidental SP allocation. Mirrors RPG convention. One-tap was considered and rejected — too easy to mis-invest. |
| 4 | Spirit Guide: placeholder in v0 | Spirit Guide placement is genuinely uncertain (sidebar? modal? overlay?). Placeholder preserves the slot without committing to a layout we'd regret. |
| 5 | Build comparison: out of v0, placeholder UI only | Comparison requires 2+ builds loaded simultaneously; out of scope for v0's single-class inspection model. Placeholder preserves the concept. |
| 6 | Cross-season identity: ON for v0 | This is the primary analytical value proposition. P2 surfaces it. |
| 7 | Color palette: neutral for v0 | Class-specific theming (fire = red, water = blue) is a v1 polish item. Neutral keeps the UI readable without per-archetype CSS work. |
| 8 | Sharing: stub URL scheme + disabled "Share Build" button | Button is visible so design intent is preserved. URL encoding is implemented so it can be activated in v1 without a refactor. Full social features (community feed, doppelganger search) are deferred. |

## Two-Page Architecture

### Page 1: Loadout (`/`)

The character inspection page. Focused on a single class at a time.

**Sections (desktop: single scroll; mobile: stacked):**

```
┌─────────────────────────────────────────────────────┐
│  [Nav: Loadout | Analytics]   [Share Build] (dim)   │
├─────────────────────────────────────────────────────┤
│  Class Header                                        │
│  Name / Archetype tag / Role / Element / Energy type │
├───────────────────────┬─────────────────────────────┤
│  Skill Tree           │  Skill Detail Panel          │
│  (tier rows ×         │  (appears when node tapped)  │
│   chain columns)      │  Role, element, effects,     │
│                       │  scaling coef, SP controls   │
├───────────────────────┴─────────────────────────────┤
│  Stats Panel                                         │
│  Stat distribution radar + raw numbers               │
│  SP budget: X / 120 used                            │
├─────────────────────────────────────────────────────┤
│  Gear Grid (10 slots, all placeholder in v0)         │
├─────────────────────────────────────────────────────┤
│  Spirit Guide (placeholder section)                  │
├─────────────────────────────────────────────────────┤
│  Action Bar: [Reset] [Save] [Share Build ⊘]         │
└─────────────────────────────────────────────────────┘
```

**Skill Tree layout:**
- Rows = Tiers (T1, T2, T3, T4)
- Columns = Chains (chain_A, chain_B, chain_C, chain_D)
- Node states: locked, available, invested, selected
- Unlock rules (B6 spec): T2 requires T1 ≥ 3 SP invested; T3 requires T1+T2 ≥ 5; T4 requires T1+T2+T3 ≥ 8
- Per-skill cap: 15 SP. Total cap: 120 SP.
- Two-step: tap node → Detail Panel slides in → "Invest 1 SP" button

**Mobile layout:**
- Tabs: Skill Tree | Stats | Gear | Spirit Guide
- Swipe between tabs
- Detail Panel: bottom sheet (slides up from bottom)

---

### Page 2: Analytics (`/analytics`)

Cross-season analytics dashboard. Aggregates all data in `data/` directory. Surfaces patterns across every season we have class JSON for.

**Sections:**

1. **Summary Cards** — Total seasons, total classes, unique archetypes, total skill nodes, avg balance modifier, mean |modifier − 1.0|
2. **Archetype Distribution** (bar chart) — How many classes per archetype across all seasons
3. **Element Distribution** (pie/donut chart) — dominant_element breakdown
4. **Balance Modifier Spread** (scatter/box chart) — modifier range per archetype; which archetypes are easier to balance
5. **Stat Allocation Radar** — Average str/dex/int/wis/vit by archetype
6. **Convergence Iterations by Archetype** (bar chart) — Which archetypes converge faster
7. **Season Timeline** (line chart) — avg modifier per season, chronological; shows engine improvement over time
8. **Skill Tree Composition** (stacked bar) — T1/T2/T3/T4 counts per archetype
9. **Anchor Diversity** (list/heat map) — Which seasonal anchors have been used; coverage

## Sharing as North-Star

The "I body-swapped into a hybrid slime-geomancer at L23 — who else is running this build?" use case drives v1+ design.

### URL Encoding Scheme (stub in v0, active in v1)

```
/loadout?class=<class_id>&season=<season_id>&build=<base64-encoded-ranks>
```

`build` parameter encodes per-skill SP allocations as a compact array, base64-encoded:

```
build = btoa(JSON.stringify(skills.map(s => s.currentRank)))
// e.g. "WzMsMSwwLDIsMCwxLDMsMCwwLDAsMCwwLDAsMF0="
```

Decoding on load: if `?build=` is present in URL, parse and apply to the class specified by `class=` + `season=`. If class isn't available locally, show an error placeholder.

### Build Comparison (deferred to v0.5/v1)

Placeholder UI: disabled "Compare Build" button in ActionBar. When active: split-panel showing two builds side by side, delta highlights on skills where allocation differs, radar overlay for stat comparison.

### Community Use Cases (v1+)

- Post build URL to Discord/Reddit → anyone can inspect your exact SP allocation
- "Find builds like mine" query: match on archetype_tag + dominant_element + top-3 invested skills
- Doppelganger prep: before body-swap, query for existing builds in target class
- Post-body-swap orientation: load build URL from someone who's been in this class before

## Data Sources

See `design/data-model.md` for field-level detail.

**What we consume:**
- Class JSON: id, archetype_tag, role_orientation, dominant_element, energy_type, range_profile, stat_distribution
- Skill fields: id, role, canonical_element, effect_category, effects, tier, chain_id, chain_position, parent_skill_ids, scaling_coefficient, energy_cost, cooldown_seconds
- balance_metadata: final_modifier, convergence_iterations, converged, actual_winrate
- Season manifest: season_id, generated_at, anchor (id, name, category), season_theme_element

**What we ignore in v0:**
- name / title_completion / flavor_text (null in most current classes; display id as fallback)
- color_palette (deferred to v1 theming)
- Gear JSON (slots are placeholders)
- fights.jsonl / trial.json (performance data, not consumed)
- gauntlet_results detail (only final_modifier and convergence_iterations surface in analytics)

**How to add a new season:**
1. Create `data/<season_id>/classes/` directory
2. Copy class JSONs and manifest.json from the engine output
3. `useSeasonData` hook discovers all seasons by reading the data index at build time (via Vite glob import)
4. Page 2 analytics automatically incorporates the new season on next build

## Open Design Questions

These are explicitly deferred — not forgotten. They should be addressed before v1 ships.

| Question | Status | Notes |
|----------|--------|-------|
| Spirit Guide placement | Deferred to v1 | Sidebar? Modal triggered from ActionBar? Persistent panel? Needs playtesting. |
| Build comparison UI | Deferred to v0.5 | Split-panel vs. overlay diff vs. table. Needs more than one class loaded simultaneously. |
| Color theming by archetype | Deferred to v1 | fire=red, water=blue, etc. Needs design pass to avoid accessibility issues. |
| Cross-season time-series granularity | Deferred | Per-season averages are straightforward. Per-archetype-per-season requires more data (≥3 classes per archetype per season). |
| Sharing: community backend | Deferred to v1+ | URL sharing works without a backend. Doppelganger search requires a query API or static index. |
| Mobile swipe gesture library | Deferred | Native CSS scroll snapping may be sufficient; evaluate at v0.5. |
